# Product Requirements

Last updated: 2026-02-16

## Product Goal

Приложение управляет жизненным циклом видео из внешних источников:

1. админ парсит видео из внешних сайтов;
2. админ проверяет и при необходимости редактирует данные;
3. админ публикует выбранные видео;
4. опубликованные видео попадают в отдельный публичный контур;
5. пользователи смотрят только публичные видео.

## Users

### Admin

Использует систему для:

- парсинга категорий и отдельных видео;
- проверки распарсенных данных;
- управления raw/canonical тегами и категориями;
- публикации и скрытия видео;
- поддержки актуальности опубликованного каталога.

### Public Viewer

Использует только публичный сайт для:

- просмотра списка видео;
- открытия карточки конкретного видео;
- воспроизведения видео через player/direct links, которые уже были подготовлены системой.

## Product Areas

### 1. Parsing

Система умеет парсить разные сайты через стратегии.
Каждая стратегия может иметь свою внутреннюю механику, но на выходе должна приводить данные видео к единому контракту.

### 2. Governance

Сырые теги и другие внешние сущности не считаются готовыми к публикации, пока админ не приведет их к каноничному виду или не проигнорирует.

### 3. Publishing

Опубликованное видео хранится в отдельной таблице как snapshot.
Это отдельный слой относительно parsed-данных.

### 4. Public Distribution

Публичный сайт не должен читать core напрямую при пользовательском запросе.
Для этого core отдает внешний integration API, из которого отдельный public backend собирает свою read-model.

## Key User Flows

### Flow: Parse Category

1. Админ отправляет URL категории.
2. Система извлекает список видео.
3. При необходимости система гидрирует найденные видео-страницы.
4. Parsed-видео сохраняются в БД.

### Flow: Prepare for Publish

1. Админ открывает parsed-видео.
2. Проверяет название, описание, источники, теги и другие данные.
3. Закрывает unmapped raw-сущности через governance.

### Flow: Publish Video

1. Админ открывает parsed-видео.
2. Может вручную скорректировать редактируемые поля перед публикацией.
3. Система создает или обновляет запись в `published_videos`.
4. Видео становится доступным для внешней репликации.

### Flow: Public Replication

1. Отдельный public backend читает `external/public-feed`.
2. Отдельно читает `external/categories`.
3. Строит локальную public read-model.
4. Публичный фронт работает только с этим public backend.

## Non-Goals

- хранение видеофайлов у себя;
- сложная auth-система;
- runtime proxy-запросы public frontend в core;
- полная переработка legacy-модулей в текущем scope;
- автоматическая публикация видео без участия админа.

## Product Constraints

- Источники могут быть нестабильными: прямые ссылки на видео могут протухать.
- У разных сайтов могут отличаться структура данных и наличие ассетов.
- Некоторые второстепенные ассеты могут отсутствовать, но player source для успеха обязателен.
- Категории должны быть доступны public-контуру автоматически, без отдельного publish flow.

## Success Criteria

- Админ может стабильно пополнять каталог новыми видео.
- Public-контур не зависит от core во время пользовательского чтения.
- Публичная выдача не содержит raw/governance внутренних деталей.
- Архитектура позволяет добавлять новые parser strategies и новые public mirrors.
